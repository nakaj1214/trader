# CodeQL 解析

対応言語: Python, JavaScript/TypeScript, Go, Java/Kotlin, C/C++, C#, Ruby, Swift。

**スキルリソース:** リファレンスファイルとテンプレートは `{baseDir}/references/` と `{baseDir}/workflows/` にあります。実行時にこれらのファイルへのパスを解決するには `{baseDir}` を使用してください。

## クイックスタート

一般的なケース（「このコードベースの脆弱性をスキャンする」）の場合:

```bash
# 1. CodeQL がインストールされていることを確認
command -v codeql >/dev/null 2>&1 && codeql --version || echo "NOT INSTALLED"

# 2. 既存のデータベースを確認
ls -dt codeql_*.db 2>/dev/null | head -1
```

次に、以下のワークフローを使用してフルパイプラインを実行します: **データベース構築 → データ拡張の作成 → 解析の実行**。

## 使用する場面

- 深いデータフロー解析によるコードベースのセキュリティ脆弱性スキャン
- ソースコードからの CodeQL データベース構築（コンパイル言語のビルド機能を含む）
- 手続き間テイント追跡や AST/CFG 解析が必要な複雑な脆弱性の発見
- 複数のクエリパックを使用した包括的なセキュリティ監査

## 使用しない場面

- **カスタムクエリの作成** - 専用のクエリ開発スキルを使用
- **CI/CD 統合** - GitHub Actions ドキュメントを直接使用
- **簡易パターン検索** - 速度重視なら Semgrep や grep を使用
- コンパイル言語の**ビルド機能がない**場合 - 代わりに Semgrep を検討
- **単一ファイルや軽量な解析** - 単純なパターンマッチングには Semgrep の方が高速

## 拒否すべき言い訳

これらの近道は検出漏れにつながります。受け入れないでください:

- **「security-extended で十分」** - これはベースラインです。常に Trail of Bits パックとコミュニティパックがその言語で利用可能かどうかを確認してください。`security-extended` が完全に見逃すカテゴリをこれらがカバーします。
- **「データベースがビルドできたから問題ない」** - ビルドできたデータベースが正しく抽出されたとは限りません。常にステップ4（品質評価）を実行し、ファイル数を期待されるソースファイル数と比較してください。キャッシュされたビルドは有用な抽出を生成しません。
- **「標準フレームワークにはデータ拡張は不要」** - Django/Spring アプリでも、ORM コール、リクエスト解析、シェル実行をラップするカスタムラッパーがあり、CodeQL はこれをモデル化しません。拡張ワークフローをスキップすると、プロジェクト固有のコードの脆弱性を見逃すことになります。
- **「コンパイル言語に build-mode=none でよい」** - これは深刻に不完全な解析を生成します。コンパイルされたコードを通じた手続き間データフローは追跡されません。絶対的な最終手段としてのみ使用し、制限事項を明確にフラグ付けしてください。
- **「検出なし = コードは安全」** - ゼロ検出は、データベースの品質不良、モデルの欠落、または誤ったクエリパックを示している可能性があります。クリーンな結果を報告する前に調査してください。
- **「デフォルトスイートを実行すればよい」** - デフォルトスイートは CodeQL の起動方法によって異なります。結果を再現可能にするため、常にスイートを明示的に指定してください（例: `security-extended`）。

## ワークフロー選択

このスキルには3つのワークフローがあります:

| ワークフロー | 目的 |
|----------|---------|
| [build-database](workflows/build-database.md) | 3つのビルド方法を順番に試して CodeQL データベースを作成 |
| [create-data-extensions](workflows/create-data-extensions.md) | プロジェクト API のデータ拡張モデルを検出または生成 |
| [run-analysis](workflows/run-analysis.md) | ルールセットの選択、クエリの実行、結果の処理 |


### 自動検出ロジック

**ユーザーが明示的に指定した場合**（例: 「データベースを構築して」、「解析を実行して」）、そのワークフローを実行します。

**「テスト」、「スキャン」、「解析」などのデフォルトパイプライン:** 3つのワークフローを順番に実行します: ビルド → 拡張 → 解析。create-data-extensions ステップは、CodeQL がデフォルトでモデル化しないカスタムフレームワークやアノテーションを持つプロジェクトの脆弱性を発見するために重要です。

```bash
# データベースが存在するか確認
DB=$(ls -dt codeql_*.db 2>/dev/null | head -1)
if [ -n "$DB" ] && codeql resolve database -- "$DB" >/dev/null 2>&1; then
  echo "DATABASE EXISTS ($DB) - can run analysis"
else
  echo "NO DATABASE - need to build first"
fi
```

| 条件 | アクション |
|-----------|--------|
| データベースが存在しない | ビルド → 拡張 → 解析（フルパイプライン）を実行 |
| データベースが存在、拡張なし | 拡張 → 解析を実行 |
| データベースが存在、拡張も存在 | ユーザーに確認: 既存 DB で解析を実行するか、再構築するか? |
| ユーザーが「解析だけ実行」または「拡張をスキップ」と言った場合 | 解析のみ実行 |


### 決定プロンプト

不明な場合はユーザーに確認:

```
CodeQL 解析のお手伝いができます。何をされたいですか？

1. **フルスキャン（推奨）** - データベース構築、拡張作成、そして解析実行
2. **データベース構築** - このコードベースから新しい CodeQL データベースを作成
3. **データ拡張の作成** - プロジェクト API 用のカスタムソース/シンクモデルを生成
4. **解析の実行** - 既存のデータベースに対してセキュリティクエリを実行

[データベースが存在する場合: "既存のデータベースを見つけました: <DB_NAME>"]
```
