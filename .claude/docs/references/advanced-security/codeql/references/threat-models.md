# 脅威モデルリファレンス

CodeQL 解析中にアクティブにするソースカテゴリを制御します。デフォルトでは `remote` ソースのみが追跡されます。

## 利用可能なモデル

| モデル | 含まれるソース | 有効にするタイミング | 誤検知への影響 |
|-------|------------------|----------------|----------------------|
| `remote` | HTTP リクエスト、ネットワーク入力 | 常時（デフォルト）。Web サービス、API、ネットワーク向けコードをカバー。 | 低 — 最も一般的な攻撃ベクトル。 |
| `local` | コマンドライン引数、ローカルファイル | CLI ツール、バッチプロセッサ、ローカルユーザーが信頼できないデスクトップアプリ。 | 中 — CLI 引数が開発者制御の Web 専用サービスではノイズを生成。 |
| `environment` | 環境変数 | 実行時に env 変数から設定を読み取るアプリ（12-Factor アプリ、コンテナ）。起動時にのみ env を読み取り検証済み設定オブジェクトに入れるアプリではスキップ。 | 中 — 多くの env 読み取りは起動時のみの設定であり、実行時のテイントされたデータではない。 |
| `database` | データベースクエリ結果 | セカンドオーダーインジェクションシナリオ: 蓄積型 XSS、他のライターが信頼できない共有データベースからのデータ。 | 高 — ほとんどのアプリは自身のデータベースを信頼する。蓄積型/セカンドオーダー攻撃を監査する場合のみ有効化。 |
| `file` | ファイル内容 | ファイルアップロードプロセッサ、ログパーサー、ユーザー提供ファイルを受け入れる設定ファイルリーダー。 | 中 — 信頼された設定ファイルを含むすべてのファイル読み取りでトリガーされる。 |

## デフォルト動作

`--threat-models` フラグなしの場合、CodeQL は `remote` のみを使用します。これはほとんどの Web アプリケーションと API に適しています。`remote` を超えて拡張することは、アプリケーションの信頼境界がローカル入力にまで及ぶ場合に有用です。

## 使用方法

`--threat-models` フラグで追加の脅威モデルを有効化:

```bash
# Web サービス（デフォルト — remote のみ）
codeql database analyze codeql.db \
  -- codeql/python-queries

# CLI ツール — ローカルユーザーが悪意のある入力を提供可能
codeql database analyze codeql.db \
  --threat-models=remote,local \
  -- codeql/python-queries

# 信頼できないオーケストレータから env 変数を読み取るコンテナアプリ
codeql database analyze codeql.db \
  --threat-models=remote,environment \
  -- codeql/python-queries

# フルカバレッジ — すべての入力ベクトルの監査モード
codeql database analyze codeql.db \
  --threat-models=remote,local,environment,database,file \
  -- codeql/python-queries
```

複数のモデルを組み合わせることができます。追加のモデルごとに、CodeQL がテイントされたとみなすソースのセットが拡大し、カバレッジが増加しますが、誤検知が増加する可能性があります。アプリケーションの実際の脅威モデルに合致する最も狭いセットから始め、必要に応じて拡張してください。
