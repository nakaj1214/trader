# 安全でないデフォルト値の検出

設定が欠落した状態でアプリが安全でなく動作する**フェイルオープン**脆弱性を発見します。悪用可能なデフォルトと、安全にクラッシュするフェイルセキュアパターンを区別します。

- **フェイルオープン（危険）:** `SECRET = env.get('KEY') or 'default'` → アプリが弱いシークレットで動作
- **フェイルセキュア（安全）:** `SECRET = env['KEY']` → 欠落時にアプリがクラッシュ

## 使用する場面

- 本番アプリケーションの**セキュリティ監査**（認証、暗号、API セキュリティ）
- デプロイファイル、IaC テンプレート、Docker 設定の**設定レビュー**
- 環境変数の取り扱いとシークレット管理の**コードレビュー**
- ハードコードされた認証情報や弱いデフォルトの**デプロイ前チェック**

## 使用しない場面

このスキルを使用しない場面:
- テスト環境に明示的にスコープされた**テストフィクスチャ**（`test/`、`spec/`、`__tests__/` 内のファイル）
- **例/テンプレートファイル**（`.example`、`.template`、`.sample` サフィックス）
- **開発専用ツール**（開発用ローカル Docker Compose、デバッグスクリプト）
- README.md や docs/ ディレクトリ内の**ドキュメント例**
- デプロイ時に置き換えられる**ビルド時設定**
- 適切な設定なしにアプリが起動しない**欠落時クラッシュ動作**（フェイルセキュア）

迷った場合: コードパスを追跡して、アプリがデフォルトで動作するかクラッシュするかを判断してください。

## 拒否すべき言い訳

- **「開発用のデフォルトだ」** → 本番コードに到達する場合、発見事項です
- **「本番設定で上書きされる」** → 本番設定が存在することを確認; そうでなければコードレベルの脆弱性が残る
- **「適切な設定なしでは絶対に動作しない」** → コードトレースで証明; 多くのアプリは暗黙的に失敗する
- **「認証の後ろにある」** → 多層防御; セッションが侵害されれば弱いデフォルトが悪用される
- **「リリース前に修正する」** → 今記録する; 「後で」が来ることは稀

## ワークフロー

すべての潜在的な発見に対してこのワークフローに従ってください:

### 1. 検索: プロジェクト調査と安全でないデフォルトの発見

言語、フレームワーク、プロジェクトの慣例を判断します。この情報を使用して、シークレットの保管場所、シークレットの使用パターン、認証情報付きのサードパーティ統合、暗号化、その他の関連設定をさらに調査します。さらにこの情報を使用して、安全でないデフォルト設定を分析します。

**例**
`**/config/`、`**/auth/`、`**/database/`、env ファイル内のパターンを検索:
- **フォールバックシークレット:** `getenv.*\) or ['"]`, `process\.env\.[A-Z_]+ \|\| ['"]`, `ENV\.fetch.*default:`
- **ハードコードされた認証情報:** `password.*=.*['"][^'"]{8,}['"]`, `api[_-]?key.*=.*['"][^'"]+['"]`
- **弱いデフォルト:** `DEBUG.*=.*true`, `AUTH.*=.*false`, `CORS.*=.*\*`
- **暗号アルゴリズム:** セキュリティコンテキストでの `MD5|SHA1|DES|RC4|ECB`

調査結果に基づいて検索アプローチを調整してください。

テストフィクスチャや例ファイルではなく、本番到達可能なコードに焦点を当ててください。

### 2. 検証: 実際の動作
各マッチについて、コードパスを追跡して実行時の動作を理解します。

**回答すべき質問:**
- このコードはいつ実行されるか?（起動時 vs 実行時）
- 設定変数が欠落している場合、何が起こるか?
- 安全な設定を強制するバリデーションがあるか?

### 3. 確認: 本番への影響
この問題が本番に到達するかどうかを判断:

本番設定が変数を提供する場合 → 深刻度は低い（ただしコードレベルの脆弱性は残る）
本番設定が欠落しているかデフォルトを使用する場合 → 重大

### 4. 報告: 証拠付き

**レポート例:**
```
発見事項: ハードコードされた JWT シークレットのフォールバック
場所: src/auth/jwt.ts:15
パターン: const secret = process.env.JWT_SECRET || 'default';

検証: JWT_SECRET なしでアプリが起動; シークレットが42行目の jwt.sign() で使用
本番への影響: Dockerfile に JWT_SECRET が欠落
悪用方法: 攻撃者が 'default' を使用して JWT を偽造し、不正アクセスを取得
```

## クイック検証チェックリスト

**フォールバックシークレット:** `SECRET = env.get(X) or Y`
→ 検証: env 変数なしでアプリが起動するか? シークレットが暗号/認証で使用されているか?
→ スキップ: テストフィクスチャ、例ファイル

**デフォルト認証情報:** ハードコードされた `username`/`password` ペア
→ 検証: デプロイされた設定でアクティブか? 実行時のオーバーライドがないか?
→ スキップ: 無効化されたアカウント、ドキュメント例

**フェイルオープンセキュリティ:** `AUTH_REQUIRED = env.get(X, 'false')`
→ 検証: デフォルトが安全でないか (false/disabled/permissive)?
→ 安全: アプリがクラッシュするかデフォルトが安全 (true/enabled/restricted)

**弱い暗号:** セキュリティコンテキストでの MD5/SHA1/DES/RC4/ECB
→ 検証: パスワード、暗号化、トークンで使用されているか?
→ スキップ: チェックサム、セキュリティ以外のハッシュ

**許容的なアクセス:** CORS `*`、パーミッション `0777`、デフォルトで公開
→ 検証: デフォルトが不正アクセスを許可するか?
→ スキップ: 正当な理由を持って明示的に設定された許容性

**デバッグ機能:** スタックトレース、イントロスペクション、詳細エラー
→ 検証: デフォルトで有効か? レスポンスに露出しているか?
→ スキップ: ログのみ、ユーザー向けでない
