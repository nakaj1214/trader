## レビューチェックリスト

**コード品質:**
- 関心の分離は適切か？
- エラーハンドリングは適切か？
- 型安全性は確保されているか（該当する場合）？
- DRY 原則に従っているか？
- エッジケースは処理されているか？

**アーキテクチャ:**
- 設計上の判断は妥当か？
- スケーラビリティは考慮されているか？
- パフォーマンスへの影響は？
- セキュリティ上の懸念はあるか？

**テスト:**
- テストは実際のロジックをテストしているか（モックだけでなく）？
- エッジケースはカバーされているか？
- 必要な箇所に統合テストはあるか？
- すべてのテストはパスしているか？

**要件:**
- 計画のすべての要件を満たしているか？
- 実装は仕様と一致しているか？
- スコープクリープはないか？
- 破壊的変更はドキュメント化されているか？

**本番環境への準備状況:**
- マイグレーション戦略はあるか（スキーマ変更がある場合）？
- 後方互換性は考慮されているか？
- ドキュメントは完成しているか？
- 明らかなバグはないか？

## 重要なルール

**やるべきこと:**
- 実際の重大度で分類する（すべてをクリティカルにしない）
- 具体的に記述する（曖昧でなく、ファイル:行で指定する）
- 問題が重要な理由を説明する
- 良い点を認める
- 明確な判定を下す

**やってはいけないこと:**
- 確認せずに「問題なし」と言う
- 些細な指摘をクリティカルとマークする
- レビューしていないコードについてフィードバックする
- 曖昧な表現をする（「エラーハンドリングを改善」など）
- 明確な判定を避ける
