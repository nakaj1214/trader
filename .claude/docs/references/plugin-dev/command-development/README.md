# コマンド開発スキル

Claude Codeスラッシュコマンドの作成に関する包括的なガイダンス。ファイルフォーマット、フロントマターオプション、動的引数、ベストプラクティスを含みます。

## 概要

このスキルは以下の知識を提供します：
- スラッシュコマンドのファイルフォーマットと構造
- YAMLフロントマターの設定フィールド
- 動的引数（$ARGUMENTS、$1、$2など）
- @構文によるファイル参照
- !`構文によるBash実行
- コマンドの整理と名前空間
- コマンド開発のベストプラクティス
- プラグイン固有の機能（${CLAUDE_PLUGIN_ROOT}、プラグインパターン）
- プラグインコンポーネント（エージェント、スキル、フック）との統合
- バリデーションパターンとエラーハンドリング

## コマンド基本クイックリファレンス

### ファイルフォーマット

```markdown
---
description: Brief description
argument-hint: [arg1] [arg2]
allowed-tools: Read, Bash(git:*)
---

Command prompt content with:
- Arguments: $1, $2, or $ARGUMENTS
- Files: @path/to/file
- Bash: !`command here`
```

### 配置場所

- **プロジェクト**: `.claude/commands/`（チームと共有）
- **個人**: `~/.claude/commands/`（自分だけのコマンド）
- **プラグイン**: `plugin-name/commands/`（プラグイン固有）

### 主な機能

**動的引数：**
- `$ARGUMENTS` - すべての引数を単一文字列として
- `$1`, `$2`, `$3` - 位置引数

**ファイル参照：**
- `@path/to/file` - ファイル内容を含める

**Bash実行：**
- `!`command`` - 実行して出力を含める

## フロントマターフィールド クイックリファレンス

| フィールド | 目的 | 例 |
|-------|---------|---------|
| `description` | /helpでの簡潔な説明 | `"Review code for issues"` |
| `allowed-tools` | ツールアクセスを制限 | `Read, Bash(git:*)` |
| `model` | モデルを指定 | `sonnet`, `opus`, `haiku` |
| `argument-hint` | 引数を文書化 | `[pr-number] [priority]` |
| `disable-model-invocation` | 手動専用コマンド | `true` |

## 一般的なパターン

### シンプルなレビューコマンド

```markdown
---
description: Review code for issues
---

Review this code for quality and potential bugs.
```

### 引数付きコマンド

```markdown
---
description: Deploy to environment
argument-hint: [environment] [version]
---

Deploy to $1 environment using version $2
```

### ファイル参照付きコマンド

```markdown
---
description: Document file
argument-hint: [file-path]
---

Generate documentation for @$1
```

### Bash実行付きコマンド

```markdown
---
description: Show Git status
allowed-tools: Bash(git:*)
---

Current status: !`git status`
Recent commits: !`git log --oneline -5`
```

## 開発ワークフロー

1. **コマンドを設計する：**
   - 目的とスコープを定義
   - 必要な引数を決定
   - 必要なツールを特定

2. **ファイルを作成する：**
   - 適切な場所を選択
   - コマンド名の `.md` ファイルを作成
   - 基本的なプロンプトを記述

3. **フロントマターを追加する：**
   - 最小限（descriptionのみ）から開始
   - 必要に応じてフィールドを追加（allowed-toolsなど）
   - argument-hintで引数を文書化

4. **コマンドをテストする：**
   - `/command-name` で呼び出し
   - 引数が動作することを確認
   - Bash実行をチェック
   - ファイル参照をテスト

5. **改善する：**
   - プロンプトの明確さを向上
   - エッジケースを処理
   - コメントに例を追加
   - 要件を文書化

## ベストプラクティスまとめ

1. **単一責務**: 1つのコマンド、1つの明確な目的
2. **明確な説明**: `/help` で見つけやすくする
3. **引数を文書化**: 常にargument-hintを使用
4. **最小限のツール**: 最も制限的なallowed-toolsを使用
5. **徹底的にテスト**: すべての機能が動作することを確認
6. **コメントを追加**: 複雑なロジックを説明
7. **エラーを処理**: 引数やファイルの欠落を考慮
