# 中立的分析ルール

迎合（Sycophancy）による幻覚を防ぐための指示設計原則。

## 中立的な指示を出す

結論を誘導する指示ではなく、観察ベースの中立的な指示を使うこと:

```
# ❌ 悪い例: 結論を誘導している
「このコードのバグを探せ」
「パフォーマンスの問題を見つけて」
「セキュリティホールを指摘して」

# ✅ 良い例: 中立的な観察を求めている
「このコードのロジックを追って、観察結果を報告せよ」
「実行パスをトレースし、各ステップの入出力を記録せよ」
「このコードの動作を説明し、想定と異なる箇所があれば指摘せよ」
```

## 三者構造で精度を上げる

重要な判断では、1つの視点に依存しない:

| 役割 | 目的 | 傾向 |
|------|------|------|
| **列挙役** | 問題を幅広く拾う | 過剰に問題を挙げる（偽陽性多め） |
| **反証役** | 列挙された問題を検証・反駁する | 根拠のないものを潰す |
| **審判役** | 最終判定を下す | 証拠に基づく冷静な判断 |

サブエージェントを使って三者構造を実現できる:

```
1. Explore エージェント（列挙役）: 問題候補をすべて列挙
2. 別の Explore エージェント（反証役）: 各問題候補を検証・反論
3. メインスレッド（審判役）: 両方の結果を見て最終判定
```

## 自分の出力を疑う

- 「求められた成果」を出そうとして無理に捻り出していないか自問する
- 証拠なく「問題がある」「問題がない」と断言しない
- 不確実なときは「確認が必要」と明示する
