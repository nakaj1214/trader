# Create Proposal — 詳細手順

`docs/implement/prompt.md`（ユーザーの自由な依頼文）を読み込み、
構造化した `docs/implement/proposal.md`（実装要件書）を生成する。

品質チェック（7項目）を実施し、基準を満たさない場合はリライトする。
create-plan とは独立しており、本スキル単体で完結する。

---

## Fixed Files

| File | Role |
|------|------|
| `docs/implement/prompt.md` | Input: ユーザーの依頼文（読み取り専用） |
| `docs/implement/proposal.md` | Output: 構造化された実装要件書 |

---

## Step 1: prompt.md を読む

```
docs/implement/prompt.md
```

ファイルが存在しなければ「prompt.md が見つかりません」と報告して終了。

---

## Step 2: コードベース調査

prompt.md に記載された画面名・機能名から、対象ファイルを特定する:

```bash
# 画面名から Blade ファイルを探す
# 例: 「在庫画面」→ stock
Glob: **/stock*.blade.php

# ルートから Controller を特定
Grep: Route::.*stock

# JS ファイルを特定
Glob: **/stock*.js
```

対象ファイルが明記されていない要件には、調査結果を補完する。

---

## Step 3: proposal.md を生成する

prompt.md の内容を以下の構造化テンプレートに当てはめて
`docs/implement/proposal.md` を生成する。

### テンプレート

```markdown
# 実装要件書

> このファイルは create-plan の入力です。
> 各要件は「具体的な Before/After」「対象ファイル」「受入条件」を含めてください。

---

## 要件一覧

### REQ-001: {要件タイトル}

- **画面**: {画面名}
- **対象ファイル**: {ファイル名（わかる範囲で）}
- **Before（現状）**: {現在の動作・表示}
- **After（期待）**: {修正後の動作・表示}
- **受入条件**: {これを満たせば完了}
- **備考**: {補足情報}

### REQ-002: {要件タイトル}
...

---

## 繰り返し失敗している要件

> 過去に修正を試みたが解決しなかった要件。
> 根本原因の仮説と、前回と異なるアプローチを記載すること。

### RETRY-001: {要件タイトル}（N回目の試行）

- **画面**: {画面名}
- **対象ファイル**: {ファイル名}
- **Before（現状）**: {現在の動作}
- **After（期待）**: {修正後の動作}
- **前回の試行**:
  - 変更内容: {何を変更したか}
  - 失敗理由: {なぜ直らなかったか（わかる範囲で）}
- **今回のアプローチ**: {前回と異なる方法}
- **受入条件**: {具体的な確認方法}
- **調査が必要な場合**: `systematic-debugging` スキルで原因特定を先に行う

---

## 完了済み・保留中

> 完了した要件はここに移動してください（コメントアウトではなく）。

### 完了済み
- {完了した要件を箇条書き}

### 保留中（今回のスコープ外）
- {今は対応しない要件を箇条書き}
```

### 生成ルール

1. **元の要件を勝手に削除しない** — prompt.md のすべての要件をテンプレートに移行する
2. **コメントアウトされた要件** は「完了済み」か「保留中」に分類する
   - 判断できない場合はユーザーに確認する
3. **繰り返し失敗パターン** を検出したら RETRY セクションに移動する
   - 検出パターン: `改善されていない`, `N回目`, `直っていない`, `まだ動かない`, `前回と同じ`
   - ユーザーに「前回何を変更したか覚えていますか？」と質問する
4. **曖昧な記述** は推測で具体化した上で `[要確認]` タグを付ける
   ```
   - **After（期待）**: ボタンの高さを td と同じにする [要確認: 具体的な px 値は？]
   ```
5. **対象ファイルが不明** な場合は Step 2 の調査結果で補完する

---

## Step 4: 品質チェック（7項目）

生成した proposal.md に対して品質チェックを実施する。
各項目を **PASS / FAIL** で判定する。

### チェック項目

#### 1. 繰り返し失敗パターン（CRITICAL）

「改善されていない」「N回目」等の記述がある場合 **即 FAIL**。
RETRY セクションに移動し、前回の試行と別アプローチを記載させる。

#### 2. 具体性チェック

各要件が **何を → どう変える** の形式になっているか。
主観的な表現（「違和感がある」「見づらい」）は具体的な変更に変換する。

#### 3. 対象ファイルの明示

変更対象のファイル名または画面名が明記されているか。

#### 4. Before / After の記述

「現在の状態」と「期待する状態」の両方が書かれているか。

#### 5. 1項目1要件の原則

1つの箇条書きに複数の異なる要件が混在していないか。

#### 6. 受入条件の存在

各要件に「これができたら完了」の基準があるか。

#### 7. コメントアウトされた要件の整理

HTML コメントで囲まれた要件が整理されているか。

### 判定基準

- **3項目以上 FAIL** → リライト（Step 5 へ）
- **2項目以下 FAIL** → 警告を出して続行
- ユーザーが「チェック不要」と言った場合はスキップ

---

## Step 5: リライト（FAIL 時）

### 5-1. ユーザーへの報告

```
## proposal.md 品質チェック結果

| # | チェック項目 | 結果 | 詳細 |
|---|-------------|------|------|
| 1 | 繰り返し失敗パターン | ✅/❌ | ... |
| 2 | 具体性 | ✅/❌ | ... |
| 3 | 対象ファイル明示 | ✅/❌ | ... |
| 4 | Before/After | ✅/❌ | ... |
| 5 | 1項目1要件 | ✅/❌ | ... |
| 6 | 受入条件 | ✅/❌ | ... |
| 7 | コメント整理 | ✅/❌ | ... |

**判定: FAIL（N/7）— proposal.md のリライトが必要です。**
```

### 5-2. 該当箇所をリライトして proposal.md を上書きする

元の依頼文は `docs/implement/prompt.md` に残っているためバックアップ不要。

---

## Step 6: ユーザー確認

proposal.md の内容をユーザーに提示し、以下を確認する:

1. `[要確認]` タグの付いた項目
2. 「完了済み」「保留中」の分類が正しいか
3. 繰り返し失敗している要件の「前回の試行」情報
4. 追加・削除すべき要件はあるか

---

## Step 7: 完了報告

```
## 要件書作成完了

### 作成ファイル
- docs/implement/proposal.md

### 品質チェック
- PASS: N/7
- 要確認タグ: M件

### 次のステップ
- 計画書を作成する場合は `/create-plan` を実行してください
```

**本スキルはここで終了する。create-plan は自動実行しない。**

---

## 判定の柔軟性

- 要件が1〜2個しかない場合はチェック項目 5, 7 はスキップする
- バグ修正のみの依頼は Before/After が自然に含まれるため、チェック 4 は緩く判定する
- prompt.md が既に構造化されている場合は、テンプレート変換をスキップして品質チェックのみ実施する
