# Error Doc Fix — 詳細手順

## Fixed Files

| File | Role |
|------|------|
| `docs/error/error.md` | Input/Output: エラー報告ドキュメント |

---

## Step 1: Read Error Document

Read `docs/error/error.md`.

ファイルが存在しない場合は停止し、ユーザーに通知する:
> `docs/error/error.md` が見つかりません。エラーをファイルに記載してください。

`<!-- -->` でコメントアウトされたエラーは解決済みとしてスキップする。
非コメント行のエラーのみを対象とする。

---

## Step 2: Parse Errors

各エラーから以下の情報を抽出する:

- **発生場所**: ファイルパス、URL、コマンド
- **エラーメッセージ**: 完全なエラー出力
- **再現手順**: 何をしたらエラーが出たか
- **環境**: ローカル / CI / デプロイ先

形式が不完全な場合は、可能な範囲で情報を推定し、不足分はユーザーに確認する。

---

## Step 3: Investigate Each Error

各エラーについて以下の手順で原因を特定する:

1. **発生場所のファイルを読む**: エラーが発生したファイルの該当箇所を確認
2. **エラーメッセージを分析**: スタックトレース、エラーコード、メッセージから原因を推定
3. **関連ファイルを調査**: import 先、呼び出し元、設定ファイル等を確認
4. **証拠を収集**: verify-before-fix の原則に従い、推測ではなく証拠に基づく

原因が不明な場合は `/systematic-debugging` に委譲する。

---

## Step 4: Fix Each Error

原因が特定できたエラーについて修正を実施する:

1. **修正方針を立てる**: 根本原因に対する修正（一時的な回避策は避ける）
2. **最小限の変更**: 影響範囲を最小化する
3. **修正を実施**: Edit ツールで該当箇所を修正

---

## Step 5: Verify Fixes

各修正について検証を実施する:

- テストがある場合: テストを実行
- テストがない場合: エラーの再現手順を実行し、エラーが解消されたことを確認
- ビルドが必要な場合: ビルドを実行

検証に失敗した場合は Step 3 に戻る。
2回失敗した場合は `/fix-escalation` に委譲する。

---

## Step 6: Update Error Document

修正・検証が完了したエラーを `docs/error/error.md` でコメントアウトする:

```markdown
<!-- 解決済み (YYYY-MM-DD)
## エラー: {元のエラー内容}
...
-->
```

---

## Step 7: Report Results

ユーザーに結果を報告する:

```
## エラー修正レポート

### 解決したエラー

| # | エラー | 原因 | 修正内容 | 検証結果 |
|---|--------|------|---------|---------|
| 1 | {summary} | {cause} | {fix} | OK |

### 未解決のエラー（あれば）

| # | エラー | 理由 |
|---|--------|------|
| 1 | {summary} | {why not fixed} |

### 変更ファイル
- {file path}: {what changed}
```

---

## Notes

- 1エラーずつ順番に処理する（並列修正は副作用リスクが高い）
- 修正が他のエラーに影響する可能性がある場合は先に依存関係を確認する
- error.md に記載されていない新しいエラーが発生した場合は error.md に追記する
- セキュリティ関連のエラーは修正後にセキュリティルールのチェックリストを実行する
