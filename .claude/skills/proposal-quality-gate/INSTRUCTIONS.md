# Proposal Quality Gate — 詳細手順

create-plan の前段として `docs/implement/proposal.md` の品質を検査し、
計画作成が失敗しやすい記述パターンを検出したら、計画ではなく proposal 自体をリライトするスキル。

---

## トリガー条件

ユーザーが `create-plan` `/create-plan` 「計画を作成」「plan作成」と言ったとき、
**create-plan を実行する前に**このスキルが起動する。

---

## 実行フロー

```
proposal.md を読む
    ↓
品質チェック（7項目）
    ↓
 ┌─ PASS → create-plan にそのまま進む
 └─ FAIL → proposal.md をリライトしてユーザーに確認
              ↓
           ユーザー承認後 → create-plan に進む
```

---

## Step 1: proposal.md を読む

```
docs/implement/proposal.md
```

ファイルが存在しなければ「proposal.md が見つかりません」と報告して終了。

---

## Step 2: 品質チェック（7項目）

各項目を **PASS / FAIL** で判定する。
**3項目以上 FAIL** の場合、リライトフェーズに進む。
2項目以下の FAIL は警告を出しつつ create-plan に進んでよい。

### チェック項目

#### 1. 繰り返し失敗パターン（CRITICAL）

「改善されていない」「N回目」「直っていない」のような記述がある場合 **即 FAIL**。
この記述があるということは、前回の計画→実装で解決できなかった問題が残っている。

**検出パターン:**
- `改善されていない`
- `N回目` （Nは数字）
- `直っていない`
- `まだ動かない`
- `前回と同じ`

**リライト時の対応:**
繰り返し失敗している項目には以下を追記させる:
- 前回何を試したか（変更したファイルと内容）
- なぜ失敗したと思うか（根本原因の仮説）
- 今回試すべき別のアプローチ

#### 2. 具体性チェック

各要件が **何を → どう変える** の形式になっているか。

**FAIL の例:**
```
キャンセルボタンがtd要素よりサイズが小さいせいでデザインに違和感を覚える
```
→ 「違和感」は主観的で、Claude が何をすべきかわからない。

**PASS の例:**
```
キャンセルボタンの高さを td 要素と同じ高さ（100%）にする。
padding を削除して td 内にフィットさせる。
```

#### 3. 対象ファイルの明示

変更対象の Blade ファイル、JS ファイル、CSS ファイル、Controller 等が
明記されているか。パスまでは不要だが、少なくともファイル名や画面名が必要。

**FAIL の例:**
```
マウスホバー時のcssをすべて削除
```
→ 「すべて」がどのファイルを指すか不明。

**PASS の例:**
```
在庫画面（stock/index.blade.php）のテーブル行ホバー時の
背景色変更CSSを削除する
```

#### 4. Before / After の記述

UI 変更やバグ修正で「現在の状態」と「期待する状態」の両方が書かれているか。

**FAIL の例:**
```
ライン: y ではなく yライン となるように修正
```
→ Before は書かれているが、どのファイルのどの箇所かが不明。

**PASS の例:**
```
Before: 在庫画面の納品先欄に「y」と表示される
After: 「yライン」と表示される
対象: stock/index.blade.php の納品先カラム
```

#### 5. 1項目1要件の原則

1つの箇条書きに複数の異なる要件が混在していないか。

**FAIL の例:**
```
・商社別一覧へのリンクボタンをaタグに変更
・商社別一覧への遷移ではなく、発注依頼書等の確認ページに飛ばすように変更
・商社別一覧と発注品番一覧は特定の製造課でのみ表示予定なのでblade構文で制御
```
→ 3つの異なる変更が「商社別一覧」という括りで混在。それぞれ独立した要件にすべき。

#### 6. 受入条件の存在

各要件に「これができたら完了」の基準があるか。
ない場合、Claude が「完了」と判断するタイミングが曖昧になる。

#### 7. コメントアウトされた要件の整理

HTML コメント `<!-- ... -->` で囲まれた要件が大量にある場合 FAIL。
完了済みなら削除し、未着手なら有効化すべき。
コメントアウトの意図が不明だと、Claude がスコープを誤解する。

---

## Step 3: リライト（FAIL 時）

### 3-1. ユーザーへの報告

品質チェック結果を日本語で報告する:

```
## proposal.md 品質チェック結果

| # | チェック項目 | 結果 | 詳細 |
|---|-------------|------|------|
| 1 | 繰り返し失敗パターン | ❌ FAIL | 2件検出（在庫画面スクロール、列選択色） |
| 2 | 具体性 | ❌ FAIL | 5件の要件が曖昧 |
| 3 | 対象ファイル明示 | ⚠️ WARN | 3件でファイル未指定 |
| 4 | Before/After | ❌ FAIL | 7件中4件で欠落 |
| 5 | 1項目1要件 | ✅ PASS | |
| 6 | 受入条件 | ❌ FAIL | 全件で欠落 |
| 7 | コメント整理 | ❌ FAIL | 40行以上のコメントアウト |

**判定: FAIL（5/7）— proposal.md のリライトが必要です。**

計画を立てる前に proposal.md を構造化します。
```

### 3-2. proposal.md を構造化テンプレートでリライト

以下のテンプレートに当てはめて `docs/implement/proposal.md` を**上書き**する。
元の内容は `docs/implement/proposal_original_YYYYMMDD.md` にバックアップする。

#### テンプレート

```markdown
# 実装要件書

> このファイルは create-plan の入力です。
> 各要件は「具体的な Before/After」「対象ファイル」「受入条件」を含めてください。

---

## 要件一覧

### REQ-001: {要件タイトル}

- **画面**: {画面名}
- **対象ファイル**: {ファイル名（わかる範囲で）}
- **Before（現状）**: {現在の動作・表示}
- **After（期待）**: {修正後の動作・表示}
- **受入条件**: {これを満たせば完了}
- **備考**: {補足情報}

### REQ-002: {要件タイトル}
...

---

## 繰り返し失敗している要件

> 過去に修正を試みたが解決しなかった要件。
> 根本原因の仮説と、前回と異なるアプローチを記載すること。

### RETRY-001: {要件タイトル}（N回目の試行）

- **画面**: {画面名}
- **対象ファイル**: {ファイル名}
- **Before（現状）**: {現在の動作}
- **After（期待）**: {修正後の動作}
- **前回の試行**:
  - 変更内容: {何を変更したか}
  - 失敗理由: {なぜ直らなかったか（わかる範囲で）}
- **今回のアプローチ**: {前回と異なる方法}
- **受入条件**: {具体的な確認方法}
- **調査が必要な場合**: `systematic-debugging` スキルで原因特定を先に行う

---

## 完了済み・保留中

> 完了した要件はここに移動してください（コメントアウトではなく）。

### 完了済み
- {完了した要件を箇条書き}

### 保留中（今回のスコープ外）
- {今は対応しない要件を箇条書き}
```

### 3-3. リライトのルール

1. **元の要件を勝手に削除しない** — すべての要件をテンプレートに移行する
2. **コメントアウトされた要件** は「完了済み」か「保留中」に分類する
   - 判断できない場合はユーザーに確認する
3. **繰り返し失敗している要件** は RETRY セクションに移動する
   - 「前回の試行」欄は空でもよいが、セクション自体は作る
   - ユーザーに「前回何を変更したか覚えていますか？」と質問する
4. **曖昧な記述** は推測で具体化した上で `[要確認]` タグを付ける
   ```
   - **After（期待）**: ボタンの高さを td と同じにする [要確認: 具体的な px 値は？]
   ```
5. **対象ファイルが不明** な場合はコードベースを調査して推測する
   - Explore エージェントを使い、画面名からファイルを特定する

### 3-4. ユーザー確認

リライト後、ユーザーに以下を確認する:

1. `[要確認]` タグの付いた項目の確認
2. 「完了済み」「保留中」の分類が正しいか
3. 繰り返し失敗している要件の「前回の試行」情報
4. 追加・削除すべき要件はあるか

ユーザーが承認したら、create-plan スキルに進む。

---

## Step 4: create-plan への引き継ぎ

品質チェックが PASS した場合、またはリライト後にユーザーが承認した場合:

1. 「proposal.md の品質チェックを通過しました。create-plan を実行します。」と報告
2. create-plan スキルを呼び出す

---

## 判定の柔軟性

- ユーザーが「チェック不要、そのまま計画して」と言った場合はスキップする
- 要件が1〜2個しかない場合はチェック項目 5, 7 はスキップする
- バグ修正のみの proposal は Before/After が自然に含まれるため、チェック 4 は緩く判定する

---

## コードベース調査のヒント

対象ファイルを特定するための調査パターン:

```bash
# 画面名から Blade ファイルを探す
# 例: 「在庫画面」→ stock
Glob: **/stock*.blade.php

# ルートから Controller を特定
Grep: Route::.*stock

# JS ファイルを特定
Glob: **/stock*.js
```
