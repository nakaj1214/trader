# 計画の概要・実装内容

提示の計画では、現在の「trader」プロジェクトに以下の機能拡張を追加し、スクリーニング精度向上と通知の可視化を図ろうとしています。主な追加項目は **フェーズA**（Golden Cross フィルター＋Bollinger Band スコア）、**フェーズC**（ADX トレンド強度スコア）、**フェーズB**（Slack通知に価格チャート画像を添付）です。これらに伴い、`src/screener.py`、`src/notifier.py`、`src/main.py`、`config.yaml`、およびテストファイルの修正・追加が提案されています。

### フェーズA: スクリーナー強化（Golden Cross + Bollinger）

- **Golden Cross フィルター:** 計画では `compute_indicators()` で50日/200日移動平均を計算し、`golden_cross` フラグを追加します。現行の `compute_indicators()`（【15†L140-L148】）は 14 日未満で空辞書を返し、返り値には`golden_cross`は含まれていません。これを実装するには、十分なデータ（日数設定を250営業日以上）を取得する必要があります。計画通り `lookback_days: 252` に設定を変更すれば、過去200営業日のデータが得られます。`golden_cross` を `None/1.0/0.0` の三値で返す設計は妥当で、`None`（データ不足時）を「フィルターしない」、`0.0`（デスクロス＝下降トレンド）の銘柄をスコア０にするという方針も筋が通っています。

- **Bollinger Band スコア:** 同様に `compute_indicators()` に 20日・±2σ のボリンジャーバンドを計算し、価格がバンド上限付近にあるほど高得点とする `bb_pos` 指標を追加します。`ta` ライブラリが既に使われているため、`ta.volatility.BollingerBands` の利用は可能です。データ期間が20未満の場合はインジケータがNaNになる恐れがありますが、現状の実装ではデータ長14以上で計算を実行しているので、15営業日以上あれば計算自体は走ります（ただしNaNリスクはテスト等で検証推奨）。`score_stock()` では `bb_pos` を取得し、重み `bb_position` と掛け合わせる設計です。提案のウェイト（例えば `bb_position: 0.10`）は適切に組み込めば、既存の `price_change_1m` や `rsi_score` などとのバランスをとれます。

- **既存スコアリングとの統合:** 現行の `score_stock(indicators, weights)`（【16†L178-L186】）は単に指標に重みを掛けて合計していますが、Golden Cross フィルター導入のため署名を `score_stock(indicators, config)` に変更する必要があります。この変更により、`config` 内で `use_golden_cross_filter` のオンオフやウェイトを取得します。計画ではデスクロス時にスコアを0に「ハードフィルター」するロジックが示されています。これに伴い、既存のテスト（例：`test_score_stock_weights()`【38†L37-L45】など）は第二引数を単なるウェイト辞書から `{screening: {weights: …}}` 形式に修正する必要があります。計画の通りテストを更新すれば互換性を保てます。

- **テストと互換性:** 既存テスト `test_score_stock_weights` や `test_score_stock_with_fifty2w` では現状 `score_stock(indicators, weights)` としているため、シグネチャ変更後は呼び出しを修正する必要があります【38†L37-L45】【39†L143-L153】。また、新たに `golden_cross is None`（データ不足時）・`golden_cross == 0.0`（デスクロス）のケースを検証するテストも追加すべきです。計画では「Golden Cross ありなし・データ不足のテストケース追加」が指摘されていますが、具体的には例えば `golden_cross=None` でもスコアは通常通り計算されること、`use_golden_cross_filter=False` でデスクロスでもスコア0にならないこと、などを確認するテストを実装すると安心です。

### フェーズC: ADX トレンド強度フィルター

- **ADX の導入:** ADX（Average Directional Index）でトレンド強度をスコア化します。計画の通り `ta.trend.ADXIndicator` を用い、14日AD値を算出し、`adx_score = min(adx/50, 1.0)` として指標に追加します。ADXは通常「25以上でトレンドあり、50以上で強いトレンド」と言われます。ここでは線形にスケールしているため、最大重み（0.10程度）をかけても過度ではなく、弱いトレンド銘柄の減点に寄与します。もしADXデータが不足すると `.adx()` がNaNを返すので、計画通り0.5のデフォルトを用いて中間スコア扱いとする方法は妥当です。`score_stock()` 内では単に `adx_score = indicators.get("adx_score", 0.5)` と取り出し、重みを乗じれば良いでしょう。

- **ウェイト調整:** ADX重み（計画例では0.10）が加わると、既存ウェイトの合計は1.0を超えるため、他の指標（RSI/MACD/52週高値）の重みをやや減らす必要があります。計画では最終的に価格変動0.20、出来高0.15、RSI0.15、MACD0.15、52週0.15、BB0.10、ADX0.10で合計1.0としています。このバランスは良い線ですが、実際のスクリーニング効果を見て細かく調整する余地はあります。

### フェーズB: Slack 通知にチャート画像添付

- **チャート生成:** 週次通知に株価チャート画像（終値+SMA+BB+出来高）を添付し、視覚的な検証を可能にします。`src/chart_builder.py` 新設案は妥当です。`yfinance` を使って過去データを取得し、`matplotlib`（要 `requirements.txt` 追加）でPNG出力するという設計は一般的です。図の内容（SMA20/50/200, Bollingerバンド, 出来高付き）の要件も適切です。注意点として、Matplotlib はデフォルトで GUI バックエンドを使うことがあるため、CI環境では `Agg` などの非対話型バックエンド設定が必要です。また、データ数が20未満でもエラーとしないことが計画で言及されていますが、実装時にはBollingerやSMAの計算時にNaN回避や警告出力を入れておくと安全です。

- **Slackへの送信:** 現状 `src/notifier.py` はWebhookでテキストのみ送信しており、ファイルアップロード機能はありません。計画では**Slack Bot Token（`xoxb-...`）**を使って`files.upload` API を呼び出す方法に切り替えています。これは正しい判断で、Webhookだけではファイル添付できません。実装面では、環境変数 `SLACK_BOT_TOKEN` を`.env`およびGitHub Secretsに追加し、Bot TokenでSlack APIにPOSTする必要があります。Pythonでは公式の `slack_sdk`（旧slackclient）ライブラリを使うか、単純に `requests` でファイルアップロードエンドポイントにデータを送る方法があります。計画書のテスト案にもある通り、Botトークン未設定時には例外を投げずにスキップし、Slack APIが5xx系を返したら `notify()` が `False` を返すよう例外処理・返り値の設計が必要です。

- **メインからの呼び出し:** `src/main.py` では現状通知にタプル `(predictions, accuracy, config)` しか渡していません（【7†L132-L140】）。計画通り `tickers_for_chart` を追加するには呼び出しを `notify(predictions, accuracy, config, tickers_for_chart=tickers)` とします。空データ時のガード（`if not empty and "ticker" in columns`）で `None` を渡す案は堅実です。

- **テスト対応:** 新設する `tests/test_chart_builder.py` では、実際にPNGバイト列が返ることや少量データでの耐性を確認すべきです。`tests/test_notifier.py` では、「tickers=Noneのとき画像生成せず既存動作と同じ」「`notifications.slack_chart=false` ならupload呼び出さない」「Bot Token未設定時に例外出さない」「Slack APIエラー時に`notify()`がFalseを返す」といった分岐テストを加える必要があります。

### config.yaml・CIの互換性

- **設定ファイルの追加:** `config.yaml` に新たに `screening.use_golden_cross_filter`、`notifications.slack_chart`、`notifications.chart_lookback_days` などの項目が加わります。これに合わせて `.env.example` への `SLACK_BOT_TOKEN` 追記、GitHub Actions (`weekly_run.yml`) の Secrets への登録（`SLACK_BOT_TOKEN`）が必要です。READMEの「GitHub Actions 必要なSecrets」欄【6†L240-L248】には現在Webhookのみ記載されていますが、Bot Tokenも追加しましょう。

- **既存との衝突確認:** 既存のSlack通知機能は `SLACK_WEBHOOK_URL` を想定しており、チャート添付用機能は `SLACK_BOT_TOKEN` を追加するだけなので互換性は保てます。`notifications.slack.enabled` が使われていればWebhook送信は継続可能です。なお、テストやCIで追加したSecretsがない場合に失敗しないよう、環境変数チェックには注意が必要です。

- **CI実行時間:** `lookback_days: 252` に増やすことでデータ取得量が約8倍になりますが、既に1バッチ50銘柄×数バッチで数百銘柄取得し、途中で1秒スリープしている実装なので、CI（GitHub Actions）の60分制限内には収まりそうです。ただし、銘柄数や過去データの量次第では時間が増加するため、場合によってはバッチサイズやスリープを調整するか、経過ログを出す等して監視すると良いでしょう。

### 検討すべき点・改善案

- **データ不足時の扱い:** Bollinger Band や ADX 計算には十分なデータ長が必要です。現行では14日未満でスキップしていますが、SMA50/200やBBにはそれ以上の日数が必要です。`lookback_days` を252にしても、例えば銘柄が上場直後等でデータ不足の場合は指標がNaNになる可能性があります。`compute_indicators()` 内でそれらを検出し、`None`を返すか既存処理を維持するかルール化すると安全です。

- **ADXの解釈:** ADXはトレンドの強さのみを示し、方向性は示しません。上昇トレンドか下降トレンドかを見るにはADXと合わせてDI+(上昇)とDI–(下降)を見る必要があります。計画では単純にADXを高いほどスコア化しますが、上昇トレンドではない強い下降トレンド銘柄も高評価してしまいます。必要に応じてADXとMACDやSMAの向きを組み合わせてフィルタすることも考えられます。

- **Slack通知のチャネル指定:** ファイルアップロード時には対象チャンネルIDが必要です。WebhookとBot Tokenが同じチャンネルを指すようにするか、Botでチャネルを指定する設定が必要です。もし `config["slack"]["channel"]` 等にチャンネル名があれば、Botトークン使用時にもそれを参照する実装にすると一貫性があります。

- **画像生成のリソース:** Matplotlib で毎週複数銘柄のチャートを生成すると、CPUやメモリを消費します。スクリーニング上位N銘柄のみ対象にすることで枚数を絞る、画像サイズ・DPIを調整してファイルサイズを抑えるなどの工夫も検討できます。

- **AIエージェント (Antigravity) との関連:** 元記事ではTradingView画面の自動読み取りによるレポート生成が差別化点ですが、本計画では独自にデータ取得～描画しています。TradingViewのスクリーンショット解析に依存しないため独立性は高いですが、記事のコンテキスト（AIがTradingViewを操作、画像生成など）とは若干アプローチが異なります。ただし可視化や自動化の目的は共通しています。

### 実装優先度と順序

計画で示された順序（A-1 → A-2 → C → B）は理に適っています。まずスクリーナー関連（Golden Cross・Bollinger・ADX）を実装し、テストで機能検証した上で、最後に通知周り（チャート生成とSlack送信）に着手します。Slack関連はファイル添付やトークン設定など新規要素が多いため、既存のフローに影響を与えにくい段階で行うのは合理的です。

## まとめ

全体として、提案の追加機能は**スクリーニング精度向上と通知の視覚化**に貢献できる有効な内容です。Golden CrossやADX、Bollingerといったテクニカル指標を組み込むことで、上昇トレンド銘柄の選別精度を高められる可能性があります。一方で実装時にはデータ取得量増加による処理時間の増大、指標計算のための十分な過去データ確保、Slack APIの扱い（Bot トークン・ファイルアップロード）などに注意する必要があります。また、テストの更新漏れや環境変数の設定も重要です。提示計画には既存テスト修正や追加テストの指摘が含まれており、この通り実装すれば互換性や信頼性を維持できます。

以上を踏まえ、計画通り段階的に実装・検証を進めれば効果的な拡張となると考えられます。ただし前述したようにADXの解釈、チャート画像処理の効率化、環境変数の管理など、細部の検討・実装漏れチェックは重要です。

**参考:** 現行コードではスコアリングに価格変動率・RSI・MACDしか使っていません【16†L178-L186】が、これらにGolden Cross/ADX/BBを加える計画は合理的です。また `src/screener.score_stock` の既存のシグネチャが `(indicators, weights)` である点や、テストがそれを前提としている点は【38†L37-L45】からも確認でき、今回の変更でテスト修正が必要になります。ソースコードやテストを適切に更新し、CIで全テストを通過させれば、安全に機能追加ができる見込みです。
